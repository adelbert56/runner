# UI 商品化驗收標準

這份文件是跑者廣場的 UI 硬性驗收規則。任何版面、樣式、內容卡片、工具列或手機響應式修改，都必須先符合這份標準，不能再用單點補丁方式交差。

## 核心原則

- 商品化優先：畫面要像可公開給跑者使用的產品，不是資料表、內部測試頁或臨時公告欄。
- 系統一致優先：先維護共用設計系統，再處理單頁例外。
- 手機優先：所有主要流程必須先在手機寬度能閱讀、篩選、收藏、點擊。
- 資訊層級清楚：主要資訊、次要資訊、操作按鈕、補充資料不能互相搶版面。
- 不接受補丁式 UI：如果同類元件有不同樣式，要回頭抽成共用樣式，不是在該元件上再補一段例外 CSS。

## 設計系統要求

- 色彩：主色維持深綠；黃色只用於主要提醒或重點 CTA；避免整頁只有單一綠色變化。
- 字級：頁面標題、卡片標題、摘要、標籤、按鈕要有固定階層；卡片內不得使用英雄級大字。
- 間距：同類卡片、工具列與表單欄位使用一致 gap、padding、radius。
- 圓角：一般卡片與工具控制使用 8-10px，不使用過度圓潤的裝飾感。
- 陰影：只用於可互動或主要容器，不讓每個區塊都浮起來。
- 按鈕：主要 CTA、次要 CTA、圖示按鈕、文字連結要有固定樣式，不可混用。

## 內容卡片規則

跑鞋、新聞、入門知識卡片固定順序：

1. 日期或階段
2. 類別標籤
3. 完整標題
4. 跑者視角摘要
5. 來源或下一步
6. 收藏圖示

驗收重點：

- 收藏固定在右上角，不佔正文行。
- 標題要完整可讀，不得不必要截斷。
- 摘要要是整理後的跑者觀點，不要重複來源名稱或贅詞。
- 摘要不能直接貼文章前幾句；要讀完內容後整理成「這對跑者有什麼用」。
- 中文摘要正文首行使用縮排，避免卡片看起來像後台資料欄位。
- 來源放在卡片底部，不要插在摘要開頭。
- 卡片高度可以因內容略有差異，但同一列視覺密度要接近。

## 工具列規則

- 篩選、排序、顯示筆數屬於資料控制；只看收藏、分享清單屬於操作控制，兩者要分區。
- 顯示筆數使用 10 / 25 / 50 分段按鈕，不用原生下拉選單。
- 桌面版可橫向排列，但不可擠壓標籤或讓按鈕換行得像補丁。
- 手機版篩選控制單欄排列；操作按鈕可雙欄排列。
- 工具列要像產品功能區，不像後台表單。

## 賽事卡片規則

- 日期、賽事名稱、地區、難度、報名狀態、距離、報名時間、官方入口必須優先顯示。
- 狀態顏色要明確區分：報名中、即將截止、已截止、停辦、過期、尚未開報。
- 主要 CTA 固定：官方報名或賽事資訊；收藏固定圖示；行事曆與排課次要。
- 詳細資料與來源放展開區，不要塞爆主卡片。

## 練跑菜單規則

- 表單、計畫摘要、教練分析、週課表必須屬於同一套視覺語言。
- 教練分析不能像臨時補上去的區塊；要用固定卡片、固定標題、固定問答樣式。
- 手機版表單欄位不可橫向溢出，時間與配速 selector 要完整可點。
- 課表順序需符合訓練邏輯與週內節奏，不能只看排版。

## 手機驗收清單

每次 UI 修改至少檢查：

- `#races`
- `#academy`
- `#training`
- `#gear`
- `#news`

手機 viewport 建議：

- 390 x 844
- 430 x 932

必看項目：

- 無橫向溢出
- 導覽不遮住內容
- 篩選列可完整操作
- 收藏、分享、CTA 不互相擠壓
- 卡片標題與摘要可讀
- 主要內容不被固定按鈕遮住

## 桌面驗收清單

桌面 viewport 建議：

- 1440 x 1000

必看項目：

- 首屏層級清楚，不被導覽或統計卡遮擋
- 內容卡片同列密度一致
- 工具列不散亂
- 大面積留白是刻意布局，不是內容沒對齊
- 跑鞋、新聞、入門、練跑四頁視覺語言一致

## 交付標準

UI 任務完成前必須至少做到：

1. `npm run check` 通過。
2. CSS/JS 有改就更新 `site/index.html` 版本參數。
3. 用 Playwright 產出手機與桌面截圖。
4. 自己檢查上述清單，不符合就先修。
5. 若修改產生新規則，更新 `SKILLS.md` 或本文件。
