# 第二大腦索引

這裡整理跑者廣場目前最重要的實作經驗與維護入口。後續代理人接手時，先看根目錄 `SKILLS.md`，再依任務讀下列文件。

## 核心文件

- `../../SKILLS.md`：專案級代理人準則，包含資料、UI、菜單、驗證與自我迭代規則。
- `資料更新SOP.md`：自動排程、資料流程、人工待補與發布檢查。
- `本地收尾整理.md`：本機資料夾、救援資料、驗證輸出與排程觀察狀態。
- `MCP配置.md`：目前不啟用 MCP；若未來要接外部服務，先用該文件判斷是否真的需要。
- `../賽事/平台爬蟲覆蓋報告.md`：官方報名平台命中、更新與解析錯誤。
- `../賽事/資料品質報告.md`：目前資料缺漏與優先補資料方向。
- `../賽事/爬蟲追蹤計畫.md`：依開報日、截止日與賽事日期安排下一次追蹤。
- `營運儀表板.md`：整合賽事完整度、官方直連率、開報後待補、跑鞋與新聞內容量。

## 目前已沉澱的經驗

- UI 不要一直加資訊，要把資訊分層：卡片顯示必要判斷，來源、場地、費用、名額放展開區或報告。
- 手機版要實際開 viewport 看，不能只靠 CSS 推測。
- 賽事報名入口要盡量直連官方平台，避免導到強迫登入的中介頁。
- 官方直連率必須嚴格計算：只有非 `running.biji.co` 的 `registration_link` 才算官方直連；來源詳情頁與空連結不算。
- 報名日期已開始但抓不到資料，通常不是資料不存在，而是平台解析器還不夠完整。
- 開報後待補資料應該是優先修 parser 的清單，不只是人工補資料清單。
- 同一賽事可能有短名與完整副標題兩種 crawler 輸出。人工補充要對同日期的所有名稱 alias 都補齊，否則其中一個名稱仍會被 `data:quality:strict` 判定為開報後缺核心欄位。
- 上線收尾不要被原始完整度誤導。遠期尚未開報賽事會壓低原始完整度，但只要上線可用完整度、開報後待補、日期異常、內容品質和 UI layout 都正常，可以先上線並讓排程追蹤。
- 跑鞋與新聞需要定期補候選資料，否則分頁會像空功能。
- Codex 本機沙盒的 `fetch failed` 常是網路權限問題；內容來源健康度應以授權網路執行或 GitHub Actions 結果判斷。
- 早上 07:00 的天氣排程曾因 `toISOString()` 使用 UTC 日期，把台灣今天算成前一天，導致 7 天後賽事被誤判成 8 天後。日期邏輯要固定用 Asia/Taipei。
- `scripts/validate-automation-rules.mjs` 是全自動化防呆 gate，已納入 `npm run check`。它會檢查 workflow cron、auto-commit file pattern、Pages 發布來源、資料快取版本、賽事資料同步，以及 Node 腳本是否使用 `scripts/lib/time.mjs` 的台灣日期 helper。
- 練跑菜單是商品化亮點，重點是依目標賽事日期、週數、跑量與配速自動調整。
- 公開網站正確網址是 `https://adelbert56.github.io/runner/`。README 或 GitHub Pages 說明若出現舊網址，要優先修正。
- 專案收尾時可清空根目錄空的 `server.out.log` / `server.err.log` 與 `scripts/**/__pycache__`；`.venv` 保留，除非要重新安裝環境。
- `.codex-rescues/` 和 `output/playwright/` 是本機收尾證據與驗證輸出，已被 `.gitignore` 忽略；不要為了乾淨而直接刪除。

## 收尾驗證流程

1. `npm run data:refresh`：重建賽事網站資料與品質報告。
2. `npm run content:refresh`：重建跑鞋 / 新聞候選與上架資料。
3. `npm run ops:dashboard`：更新營運儀表板。
4. `npm run check`：檢查 JS 語法、內容數量、摘要品質與前端資料載入基線。
5. `node scripts/validate-automation-rules.mjs`：確認排程、快取、資料同步與台灣日期規則沒有回歸；通常已包含在 `npm run check`。
6. `uv run python -m compileall scripts`：確認 Python 爬蟲與平台 adapters 可編譯。
7. `npm run ui:layout`：產出手機與桌面五個主分頁截圖，確認沒有橫向溢出、按鈕文字溢出或賽事卡過高。
